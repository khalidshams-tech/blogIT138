Learning Blog

Weekly technical notes organized by category.

Each post follows the same professional format: Objective, Problem, What I Tried, Solution, and What I Learned.

Linux

Useful Linux Commands for IT Students

Objective
Practice terminal navigation, files, permissions, logs, and basic system checks.
Problem
Linux can feel confusing when commands, paths, and permissions are new.
What I Tried
Used commands such as pwd, ls, cd, cat, chmod, chown, systemctl, and journalctl.
Solution
Created repeatable notes and tested each command in lab scenarios.
What I Learned
Good Linux troubleshooting starts with location, permissions, service status, and logs.
Networking

DNS, IP Addresses, and Connectivity Checks

Objective
Understand basic network troubleshooting from the local device to external services.
Problem
A system may be connected but still fail to reach websites or services.
What I Tried
Checked IP settings, gateway, DNS resolution, ping results, and browser behavior.
Solution
Used a layered process: local settings, gateway, DNS, firewall, then service status.
What I Learned
Networking problems are easier when each layer is tested in order.
Cloud

Launching an AWS EC2 Instance Safely

Objective
Practice cloud server setup, IAM, VPC, EC2, S3, CloudWatch, and secure remote access basics.
Problem
Cloud servers require correct keys, security groups, regions, and cost awareness.
What I Tried
Launched EC2, created a key pair, reviewed inbound rules, studied IAM and VPC basics, and connected with SSH.
Solution
Allowed only needed access, named resources clearly, and shut down unused labs.
What I Learned
Cloud work requires both technical setup and responsible security habits.
Cybersecurity

Security Habits for New IT Professionals

Objective
Build daily habits that reduce risk in labs and real environments.
Problem
Small mistakes in passwords, updates, permissions, or access rules can create risk.
What I Tried
Reviewed MFA, least privilege, patching, password hygiene, and secure SSH access.
Solution
Used checklists for access, updates, backups, and documentation.
What I Learned
Security is not only tools; it is careful behavior repeated consistently.
Troubleshooting

How I Approach Technical Problems

Objective
Create a calm process for diagnosing IT issues.
Problem
Jumping to fixes too quickly can waste time or hide the real cause.
What I Tried
Wrote down symptoms, reproduced the issue, checked logs, and tested one change at a time.
Solution
Used a simple process: observe, isolate, test, document, and verify.
What I Learned
Strong troubleshooting is organized thinking plus good documentation.
